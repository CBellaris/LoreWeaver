# 项目计划书：LoreWeaver - LLM驱动的超大尺度文本世界观分析引擎
## 一、 项目愿景与痛点切入
**1. 项目愿景**
打造一个能够对超长篇文本（如数百万字的小说、剧本档案）进行**“全局感知、时序追踪、隐性概念挖掘”**的智能分析引擎。它能够像最硬核的“设定考据党”一样，回答诸如“提取全书地图坐标及其在历史事件中的变迁”、“推演魔力的底层本质”等需要跨越数百万字进行综合推理的终极复杂问题。

**2. 核心痛点与行业现状**
*   **传统RAG（检索增强生成）的失败**：强行将连续文本切块并检索，丢失了上下文拓扑结构；只能回答“局部事实”，面临宏观综合问题时往往“盲人摸象”。
*   **传统知识图谱的过度压缩**：将极具文学价值的复杂互动粗暴降维成“三元组（A-认识-B）”，丢失了场景细节与情感语境，导致大模型无法进行真正的阅读理解。
*   **长上下文大模型的内存遗忘**：直接塞入百万字成本极高，且模型存在“中间遗忘（Lost in the Middle）”，无法保证长尾设定的高召回率。

## 二、 核心架构哲学（Core Philosophy）
本项目彻底抛弃“存储答案”的旧思维，采用**“探照灯与档案馆”架构**：
1. **不丢失任何原始语境**：绝不通过抽取关系来替代原文。数据库的作用是“极速定位到目标书架”，最终的阅读理解交还给拥有长上下文能力的大模型去“生啃”原汁原味的文本。
2. **文本空间树（Text Spatial Tree）**：打破“互斥切块”思维，允许语义块之间重叠、嵌套，构建一个基于物理坐标（字间距）的多尺度图结构。
3. **自下而上的涌现（Bottom-up Emergence）**：不预设死板的分类，让宏观百科和隐性概念从底层文本碎片中通过算法“自然生长”出来。

## 三、 系统技术架构设计（四层模型）
### Layer 0: 绝对文本基座 (Raw Text Foundation)
*   **机制**：将千万级原始文本作为静态资源存储，不作任何删减。
*   **功能**：作为整座图谱的“金矿”。图谱中没有任何实体直接存储大段文字，而是通过精确到字符的 `[start_idx, end_idx]` 指针随时调取纯净原文。

### Layer 1: 微观跨度层 (Span Node Layer)
*   **抽取引擎**：利用LLM通读全文，框选出具有完整语义的“片段（Span）”。不强求片段独立，允许重叠。
*   **结构化属性**：每个 Span 记录摘要向量（Summary Embedding）、涉及的表面实体（人物/地点/时间）、关键词/主题标签以及文本坐标指针。
*   **坐标系图链接**：依靠数学坐标，图数据库自动生成基础拓扑边。例如，跨度坐标被包含则生成 `[:INCLUDES]` 嵌套边，坐标紧邻则生成 `[:ADJACENT_TO]` 语境边。

### Layer 2: 宏观涌现层 (Virtual Aggregation Nodes)
*   **机制**：通过定期对底层 Span 的摘要向量进行“无监督语义聚类”，将讨论同一主题（如“对风暴角的描写”）的 Span 聚拢。
*   **虚拟节点生成**：LLM 对聚类结果进行高维总结，生成“虚拟节点（如：北境地理百科）”。
*   **多级树冠**：虚拟节点向下伸出 `[:SUMMARIZES_SPAN]` 边链接到底层，向上可以再次聚类生成更高级的“世界观总纲”。最终形成倒金字塔索引结构。

### 特殊模块：概念锻造炉 (The Concept Forge)
*   *解决“未知之未知（Unknown Unknowns）”的终极武器。*
*   **抓取器**：在 Layer 1 阶段，特设开放式探针，专门抓取原文中的“反常现象（Unexplained Phenomena）”和“神棍格言/隐喻（Lore & Philosophies）”。
*   **锻造**：当大量零碎的现象被聚类时，LLM介入，为这些现象抽象出原本未被定义的隐性概念（如“魔力的辐射本质”、“神明的真面目”），并作为一个高级 Virtual Node 汇入图谱。

## 四、 极速搜索与推理工作流 (The Agentic Retrieval)
当用户抛出宏大而复杂的查询（例：“总结男主与A国关系的演变，并列出相关地理变迁”）：

1.  **混合召回（Hybrid Retrieval）**：
    *   系统不会只依赖单一路径，而是并行执行多路召回：图引导召回、向量召回、关键词/BM25 召回，以及必要的实体/章节过滤。
    *   对宏观问题，优先命中 Layer 2 的聚合节点或中心 Span 簇；对细节问题，则允许直接回落到底层 Span 检索。
2.  **候选合并与精排（Union + Reranking）**：
    *   将多路召回结果去重合并，形成较大的候选证据池。
    *   使用 Cross-Encoder 或同类 `Reranker` 对 “问题 - Span/证据块” 进行精排，剔除“语义相似但对象不对”的假阳性结果。
3.  **顺藤摸瓜与下钻（Drill-down）**：
    *   对被精排保留下来的高相关聚合节点，顺着 `[:SUMMARIZES_SPAN]` 或显式支持边向下突击，命中数十个底层 Span Nodes。
4.  **原矿提取与区间合并（Interval Merging）**：
    *   提取这些 Span 的 `[start_idx, end_idx]` 坐标。
    *   算法进行一维坐标区间的自动去重和合并（解决语义块嵌套重叠问题）。
    *   去 Layer 0 磁盘中精准切下总计约 60k - 100k Token 的高纯度连续原文本。
5.  **终极降维打击（Synthesis）**：
    *   将 问题 + 层级百科（作为思考框架） + 100k 纯原文本 提交给 长上下文大模型（GPT-4o / Claude 3.5 Sonnet）。
    *   大模型执行真正的“深度阅读理解”，输出带有原文精准引用的殿堂级分析报告。

## 五、 商业价值与应用场景
1.  **网文作者/IP架构师的神器**：彻底解决长篇连载“吃书”、伏笔遗忘问题。一键梳理人物关系网与隐秘时间线。
2.  **游戏开发与剧本杀创作**：将传统线性的文学剧本，自动化转化为包含地理拓扑、物品流通轨迹的“结构化世界观数据库”，直接赋能游戏策划。
3.  **法律长卷宗与刑侦档案分析**：跳出文娱领域，该架构完美适用于几十万字的连续案卷分析。从零散的“异常口供（现象）”中自动涌现出“隐蔽的作案动机（概念锻造炉）”。


# LoreWeaver 技术实施与演进计划书 (工程执行版)
## 零、 产品前提与工程准则
在工程实施层面，LoreWeaver 不再以“尽快复现完整终局架构”为目标，而是先围绕一个明确的产品形态做收敛：

1.  **目标用户先收缩**：首阶段只服务于**网文作者、设定党、IP 世界观整理者**，不追求法律、刑侦等高合规场景的泛化。
2.  **产品形态明确**：接受“**离线建库慢，但在线问答稳**”的形态。建库允许花数十分钟到数小时，在线问答和报告生成必须稳定、可复现、可引用。
3.  **原文是最小可信单位**：系统中任何摘要、百科、概念、报告都不是最终真相，只是导航层。最终回答必须能够回落到**原文区间**与**原文引用**。
4.  **先回答宏观问题，再追求炫技**：首版的核心价值不是“比传统 RAG 全面领先”，而是能较稳定回答**世界观级问题**，并进一步生成**带证据的设定报告**。
5.  **确定性优先**：凡是可以由程序做确定性校验的环节，不交给模型拍脑袋。尤其是文本定位、区间合并、引用去重、输出结构校验。
6.  **上层索引随时可重算**：`Layer 0` 原文绝不修改。抽取结果、向量、中心 Span、聚合边、关系边都属于可重建索引层，允许迭代中推倒重来。

## 一、 首版 MVP 的真实范围
为了避免项目一开始陷入“数据库、Agent、聚类、图谱”同时开工的泥潭，MVP 只解决下面三件事：

1.  **离线读入一本作品并建库**：支持 `.txt` 或规范化章节文本，完成切分、抽取、定位、嵌入、索引。
2.  **在线回答宏观世界观问题**：例如“这个国家与男主关系如何演变”“该作的力量体系是什么”“某势力的历史脉络如何展开”。
3.  **生成设定报告初稿**：围绕角色、势力、地理、力量体系、历史事件等主题，生成带原文引用的整理文档。

MVP 明确**不做**以下内容：

*   不做通用型知识图谱平台。
*   不做多 Agent 调度系统。
*   不在第一版强依赖 Neo4j 可视化。
*   不要求自动涌现出高质量“终极隐性概念”。
*   不承诺对任意超长文本、任意问题都完美回答。

换句话说，第一版不是“世界观分析引擎的全部形态”，而是一个**可对话、可引用、可产出设定报告的单书分析系统**。

## 二、 首版技术栈选型 (以最小复杂度落地)
MVP 阶段的核心原则是：**先把离线建库、图结构落盘和在线证据回溯跑稳，再决定是否引入更复杂的多 Agent 与更激进的自动聚类。**

*   **主语言与工程框架**：`Python`
    *   原因：文本处理生态成熟，适合快速迭代抽取、定位、检索、评估链路。
*   **结构化抽取**：`Pydantic` + `OpenAI Structured Outputs` 或 `Instructor`
    *   作用：约束抽取结果格式，避免 JSON 漂移。
*   **向量索引**：`Qdrant`
    *   作用：存储 `micro_summary/topic/evidence` 的向量与元数据，作为在线召回入口。
*   **关键词检索**：`BM25`
    *   作用：补足纯向量检索在专有名词、别称、术语、精确表述上的短板，作为混合召回的一路。
*   **精排模型（Reranker）**：Cross-Encoder 类模型，如 `BAAI/bge-reranker-v2-m3`
    *   作用：对多路召回合并后的候选 Span 做相关性精排，减少“语义相似但证据不对”的误召回。
*   **图数据库**：`Neo4j AuraDB` 或本地 `Neo4j`
    *   作用：存储 `Span / Center Span Cluster / Span Edge / Entity` 及其显式关系，作为首版图结构主存储。
    *   说明：首版就引入图，但只使用少量、可解释的节点和边类型，不追求重图谱建模。
*   **结构化元数据与构建日志**：`SQLite` 或 `DuckDB`
    *   作用：存储文档、章节、定位结果、构建任务状态、评估结果、失败重试记录等工程性数据。
    *   说明：它不再承担主图存储职责，而是作为离线流水线的辅助存储层。
*   **原始文本层**：本地 `.txt` / 标准化章节文件
    *   作用：作为唯一可信文本源，通过 `[start_idx, end_idx]` 精确切片。
*   **离线任务调度**：`asyncio` + 本地任务脚本
    *   作用：支持批量抽取、重试、断点续跑。
*   **在线服务层**：轻量 `FastAPI`（可选）
    *   作用：暴露问答与报告生成接口；若早期只做 CLI，可暂缓。
*   **模型分工**：
    *   抽取模型：`GPT-4o-mini` 或同级低成本模型。
    *   终极综合模型：长上下文、推理稳定的主模型，如 `Claude Sonnet` 或同级替代。

结论是：**MVP 采用“向量库 + 图数据库 + 轻量工程库”三层结构**，既保留 LoreWeaver 的图核心，又避免过早进入全自动知识图谱工程。

## 三、 核心数据模型 (首版就要定死)
为了让系统具备“可定位、可追溯、可重建”的特性，建议首版围绕以下对象建模：

### 1. Document
表示整本作品。

建议字段：
`document_id, title, author, source_path, total_chars, total_chapters, created_at`

### 2. Chapter
表示章节级边界，用于构建更稳定的上下文扩展，而不是只依赖纯字符邻接。

建议字段：
`chapter_id, document_id, chapter_index, chapter_title, start_idx, end_idx`

### 3. Span
表示系统内部的基础分析单元，是从候选窗口中按语义边界拆出的“微观证据候选块”。一个窗口可以产出多个 Span，Span 可以重叠或嵌套。

建议字段：
`span_id, document_id, chapter_id, window_id, span_index_in_window, window_start, window_end, span_type, micro_summary, entities, topics, salience_score, start_anchor_quote, end_anchor_quote, key_quote, span_start_idx, span_end_idx, located_text, locator_confidence, locator_status`

关键说明：

*   **不让模型直接产出最终可信坐标**。
*   模型负责给出微观 Span 的语义描述，以及 `start_anchor_quote` / `end_anchor_quote`。
*   程序负责基于双锚点回查原文，反推出 `span_start_idx` 与 `span_end_idx`。
*   若模型偶尔输出过长锚点，程序可在定位前保留靠近 Span 边界的一侧进行裁剪，避免把可用 Span 直接丢弃。
*   若原文中命中多次，则结合章节窗口、跨度长度、局部上下文、模糊匹配分数做二次消歧。

### 4. Center Span Cluster
表示首版中层图结构的核心单元。它不是完全自动涌现的虚拟百科节点，而是一个**可控聚合簇**：以一个中心 Span 为锚点，显式连接若干底层 Span，形成可检索、可追溯的主题骨架。

建议字段：
`cluster_id, document_id, center_span_id, cluster_name, cluster_type, summary, member_span_ids, confidence, status`

其中 `cluster_type` 可先限定在：
`character / faction / location / power_system / history / mystery`

补充说明：

*   `center_span_id` 是聚合簇的导航锚点，不是真理本身。
*   `member_span_ids` 是被该中心 Span 显式聚合的底层证据集合。
*   首版可以由人工先指定聚类方向，再用向量相似、共享实体、章节邻近等方式召回候选成员。

### 5. Span Edge
表示 Span 与 Span、Span 与中心 Span 之间的显式关系。

建议字段：
`edge_id, document_id, from_span_id, to_span_id, edge_type, weight, source`

其中 `edge_type` 首版建议只保留少量、可解释的类型：
`SUPPORTS / RELATED_TO / MENTIONS_ENTITY / ADJACENT_CHAPTER`

### 6. Query Evidence Pack
表示一次在线问答真正送给大模型的证据包。

建议字段：
`query_id, user_question, retrieved_span_ids, merged_intervals, evidence_text, cluster_ids, retrieval_sources, rerank_scores, token_estimate`

这个对象的意义是：把“查到了什么、切了哪些原文、为什么生成这个答案”沉淀下来，便于调试和复盘。

## 四、 离线建库流程 (首版最关键)
离线建库是 LoreWeaver 的主战场。首版不追求花哨，而追求稳定、可重复、可重跑。

### Step 1: 原文预处理
*   将整本书读入为单一连续文本，保留统一编码。
*   建立章节边界表，记录每章的 `[start_idx, end_idx]`。
*   清洗无意义噪声，如广告、排版异常、重复空白。

### Step 2: 候选窗口切分
*   先按章节切，再在章节内使用固定窗口加重叠滑动。
*   建议初始窗口大小为 `800-1500` 字，重叠 `15%-25%`。
*   这里的窗口只是“抽取输入单元”，不是最终知识单元。
*   抽取完成后，窗口可记录 `uncovered_text` 调试字段：将未被任何 located Span 覆盖的原文片段合并保存，便于人工检查这些内容是否确实不需要进入索引。

### Step 3: LLM 结构化抽取
对每个候选窗口，抽取多个微观 Span，每个 Span 包含以下信息：

*   `span_type`：对话互动、人物关系、地点设定、力量规则、事件、伏笔等可解释类型。
*   `micro_summary`：当前 Span 发生了什么或揭示了什么，应自包含主体、动作或设定点。
*   `entities`：出现的人物、地点、势力、物品等表面实体。
*   `topics`：该 Span 更像在讨论什么主题。
*   `salience_score`：重要性评分，便于后续筛选。
*   `start_anchor_quote` / `end_anchor_quote`：用于定位 Span 区间的原文双锚点。
*   `key_quote`：最能代表该 Span 核心语义的一段原文短引。

注意：`key_quote` 不是定位字段，真正的定位锚点是 `start_anchor_quote` 和 `end_anchor_quote`。

### Step 4: 引用反查与坐标落盘
*   使用 `start_anchor_quote` 和 `end_anchor_quote` 在章节原文内做精确匹配。
*   若锚点过长，先按 start/end 的边界方向裁剪到配置上限。
*   若精确匹配失败，再做局部模糊匹配。
*   若命中多个候选，优先选择顺序正确、跨度合理且匹配分最高者。
*   将最终定位得到的 `[span_start_idx, span_end_idx]` 存盘，并记录 `locator_confidence`。
*   使用 located Span 区间反算窗口内未覆盖原文，写入 `candidate_windows.uncovered_text` 作为人工抽查用的调试字段。
*   可配置保存 `located_text`，便于在 DBeaver 中直接抽查语义切分是否准确；大规模建库时可关闭。
*   定位失败的记录进入人工复核或二次重跑队列。

### Step 5: 嵌入与索引写入
*   对 `micro_summary + entities + topics`、必要时对 `key_quote` 或 `cluster_name` 生成 Embedding。
*   写入 Qdrant，保留 `document_id / chapter_id / span_id / salience_score / cluster tags` 等 payload。
*   建立 BM25 或同类倒排检索索引，为关键词召回和术语召回做准备。
*   将 `Span / Center Span Cluster / Span Edge / Entity` 写入 Neo4j。
*   同时将定位结果、构建日志、失败记录写入 SQLite 或 DuckDB。

### Step 6: 中心 Span 图构建
首版不直接上“完全自动的概念锻造炉”，而是先做**受控的中心 Span 图**：

*   先人工指定少量聚类方向，例如角色关系、势力格局、地理板块、力量体系、历史事件。
*   在高 `salience_score` 的 Span 中，为每个方向挑选一个或多个 `center_span` 作为锚点。
*   通过向量相似、共享实体、章节邻近、关键词重合等方法召回候选底层 Span。
*   由规则或人工做最后确认，将底层 Span 显式连到对应的中心 Span。
*   首版优先建立少量、可解释的边，例如 `SUPPORTS`、`RELATED_TO`、`MENTIONS_ENTITY`。

这样做的好处是：图结构从第一天就存在，但仍然保持**可控、可核查、可逐步自动化**。

## 五、 在线问答流程 (以稳为第一目标)
在线查询阶段，LoreWeaver 应该像一个谨慎的研究助理，而不是一个大而全的自主 Agent。

### Step 1: 问题分类
先将用户问题粗分为三类：

1.  **人物/势力关系类**
2.  **地理/历史演变类**
3.  **设定/力量体系类**

这个分类不需要 LangGraph，用简单分类器或 Prompt Router 即可。

### Step 2: 混合召回
在线召回阶段采用“图 + 向量 + 关键词”并行策略，而不是依赖单一路径：

*   第一层：用问题去匹配相关 `Center Span Cluster`，拿到宏观阅读框架。
*   第二层：沿着中心 Span 到底层 Span 的显式边下钻，召回一批结构化候选。
*   第三层：并行执行向量召回，找出语义上相关但未被图显式覆盖的 Span。
*   第四层：并行执行 BM25 关键词召回，补足专有名词、术语、别称、精确表述类证据。
*   第五层：对多路结果做去重合并，形成候选证据池；若图命中不足，则允许退化为“向量 + BM25”模式，保证鲁棒性。

### Step 3: Reranker 精排
候选证据池形成后，不直接送入上下文组装，而是先做一轮精排：

*   使用 Cross-Encoder 或同类 `Reranker` 对 “问题 - Span” 对进行细粒度相关性打分。
*   将图召回、向量召回、BM25 召回得到的候选统一拉到同一排序空间。
*   优先保留对象正确、主题正确、与问题约束一致的证据，压低“看起来相似但答非所问”的 Span。
*   精排后的 Top-K 才进入后续坐标扩展与区间合并。

### Step 4: 坐标扩展与区间合并
*   读取候选 Span 的定位坐标。
*   按章节边界和窗口邻近规则适度向前后扩展上下文。
*   使用 `Interval_Merger` 去重、合并重叠区间。
*   控制最终原文证据长度，不盲目塞满模型上下文。

建议首版将在线证据控制在一个保守范围内，例如 `20k-60k tokens`，而不是默认冲到 `100k`。

### Step 5: 证据包组装
送给终极模型的内容应包含三部分：

1.  用户问题
2.  相关 `Center Span Cluster` 摘要，作为阅读提纲
3.  合并后的原文证据块，并保留章节信息与引用标识

### Step 6: 最终回答与引用输出
最终回答必须满足：

*   优先输出分析结论，而非无根据猜测。
*   对关键判断附上原文引用。
*   无证据支撑的推断必须显式降级为“推测”。
*   若证据不足，应允许系统回答“当前证据不足以确认”。

这一步的目标不是“像人一样自信”，而是“像设定考据党一样谨慎”。

## 六、 设定报告生成功能 (首版核心卖点)
设定报告不是单次问答的简单拼接，而是一个更高价值的稳定输出层。

首版建议支持以下几类报告模板：

1.  **角色设定报告**
2.  **势力关系报告**
3.  **地理与地图线索报告**
4.  **力量体系 / 魔法机制报告**
5.  **时间线与重大事件报告**

报告生成流程建议为：

1.  先确定报告主题。
2.  从 `Center Span Cluster` 中召回相关聚合簇。
3.  顺着显式边下钻到底层 Span，提取高相关原文区间。
4.  生成结构化报告草稿。
5.  对每个章节结论附带原文证据或引用编号。

报告的首版验收重点不是“文笔多华丽”，而是：

*   结构清楚；
*   引用可追溯；
*   对矛盾设定能指出冲突；
*   对缺失部分能明确标记为“待确认”。

## 七、 里程碑规划 (更贴近真实开发)
### Milestone 1: 单书建库与证据问答 (v0.1)
**目标**：选择 **3-5 万字** 的中短篇样本或一部长篇的若干连续章节，跑通“离线建库 -> 在线宏观问答 -> 原文引用输出”的完整链路。

*   **Task 1.1: 文本入库与章节切分**
    *   实现原文读取、章节边界识别、统一编码清洗。
*   **Task 1.2: 结构化抽取与定位**
    *   对滑动窗口执行多 Span 结构化抽取。
    *   输出 `span_type / micro_summary / entities / topics / salience_score / start_anchor_quote / end_anchor_quote / key_quote`。
    *   通过程序回查双锚点得到可信坐标。
*   **Task 1.3: 向量与元数据双写**
    *   将摘要向量写入 Qdrant。
    *   建立 BM25 索引，保证术语和关键词召回能力。
    *   将 `Span` 与第一批图关系写入 Neo4j。
    *   将定位结果和构建日志写入 SQLite 或 DuckDB。
*   **Task 1.4: 最小图骨架验证**
    *   手动指定 `2-4` 个聚类方向。
    *   为每个方向挑选少量中心 Span，并建立第一批 `SUPPORTS / RELATED_TO` 显式边。
*   **Task 1.5: 混合召回与精排接入**
    *   跑通 `图召回 + 向量召回 + BM25` 的多路合并。
    *   接入 `Reranker`，对候选证据进行统一精排。
*   **Task 1.6: 在线证据问答**
    *   支持输入自然语言问题。
    *   系统能够优先通过中心 Span 图召回证据，再合并坐标，切出原文，给出带引用的回答。

**M1 验收标准**

*   对一组 `20-30` 个人工问题，回答中 `80%` 以上包含有效原文引用。
*   至少 `70%` 的问题能得到“基本可信”的宏观回答。
*   每次回答都能展示所使用的证据区间和原文片段。
*   至少完成 `2-4` 个方向的中心 Span 图骨架，并能在问答中实际用到。
*   混合召回与 `Reranker` 能稳定参与主链路，而不是仅作为备用模块。
*   在线单次响应时间控制在可接受范围内，例如 `15-40 秒`。

### Milestone 2: 中心 Span 聚合与报告初稿 (v0.3)
**目标**：让系统不只会答单问，还能围绕一个主题形成受控聚合簇，并整理出一份像样的设定报告。

*   **Task 2.1: 可控聚合簇生成**
    *   基于高重要性 Span 和人工指定方向扩展中心 Span。
    *   为角色、势力、地点、力量体系等形成首批聚合簇。
*   **Task 2.2: 报告模板设计**
    *   定义报告章节结构、引用格式、冲突标记格式。
*   **Task 2.3: 报告生成链路**
    *   从中心 Span 聚合簇下钻到底层 Span。
    *   生成带引用的结构化报告草稿。
*   **Task 2.4: 小规模聚类探索**
    *   在人工指定方向内尝试半自动聚类或候选排序。
    *   记录哪些方向适合进一步自动化，哪些方向仍需人工控制。

**M2 验收标准**

*   能稳定生成至少三类报告模板。
*   报告中的关键结论均可回指到原文证据。
*   聚合簇能帮助报告组织，而不是只起到装饰作用。
*   用户能明显感知到“问答系统”升级为“设定整理工具”。

### Milestone 3: 关系层与上下文扩展 (v0.6)
**目标**：增强文本空间结构与聚合探索，但仍然坚持“轻量可控”，不急着做重图谱平台。

*   **Task 3.1: 轻量关系边**
    *   建立 `chapter adjacency / span overlap / entity mention / center support` 等基础关系表。
*   **Task 3.2: 上下文扩展策略**
    *   基于章节边界、相邻窗口、共享实体做证据扩展，而不是仅靠字符阈值。
*   **Task 3.3: 聚类方向评估**
    *   比较不同方向上的聚合效果，筛选适合进入下一阶段自动化的方向。
*   **Task 3.4: 复杂问题验证**
    *   测试“关系演变”“历史阶段变化”“前因后果串联”等问题。

**M3 验收标准**

*   对复杂问题的证据覆盖率明显好于 M1。
*   扩展上下文后，引用仍然可追溯，没有大规模噪声扩散。
*   至少有一类聚合方向在半自动化下表现稳定，可进入更深入探索。

### Milestone 4: 全书级离线建库与规模优化 (v1.0)
**目标**：支持单本长篇网文的完整离线建库与稳定在线服务。

*   **Task 4.1: 异步批处理**
    *   将抽取、重试、定位、嵌入写入改为可断点续跑的离线流水线。
*   **Task 4.2: 成本与速率控制**
    *   引入批处理队列、失败重试、Token 预算、分层召回截断。
*   **Task 4.3: 图结构扩展**
    *   在已有 Neo4j 图骨架上扩展更多可验证的关系模式与可视化查询能力。

**M4 验收标准**

*   能对一部长篇作品完成完整建库。
*   在线问答与报告生成在多数常见问题上表现稳定。
*   构建成本、耗时、失败率进入可接受区间。

## 八、 评估与验收方法 (必须前置，不可后补)
LoreWeaver 这类系统最怕“Demo 看起来很强，实际不可复现”。因此评估体系必须从第一天就建立。

### 1. 问题集评估
为每本测试作品人工整理 `20-50` 个问题，覆盖：

*   角色关系
*   势力变迁
*   地理信息
*   力量体系
*   时间线事件
*   隐性设定与矛盾点

### 2. 四项核心指标
*   **答案可信度**：回答是否基本正确。
*   **引用命中率**：关键结论是否附带有效原文引用。
*   **证据覆盖率**：召回的原文是否足以支撑最终结论。
*   **单位查询成本与耗时**：每次在线回答的模型成本与等待时间是否可接受。

### 3. 召回与精排专项评估
召回和重排是 LoreWeaver 首版可靠性的关键，应单独量化：

*   **多路召回 Recall@K**：图召回、向量召回、BM25 召回各自与合并后的覆盖率表现。
*   **Union 增益**：多路合并后，相比单一路径是否显著提高有效证据覆盖。
*   **Reranker 命中率**：精排后的 Top-K 中，有多少条是真正支撑答案的核心证据。
*   **误召回压制能力**：`Reranker` 是否能显著降低“语义相似但对象错误”的候选进入最终证据包。

### 4. 图与聚合专项评估
除了问答表现，还要单独评估中心 Span 图是否真的有价值：

*   **聚合可解释性**：一个聚合簇中的成员 Span 是否真的围绕同一问题方向。
*   **边质量**：`SUPPORTS / RELATED_TO` 是否能被人工快速理解和接受。
*   **查询增益**：启用图引导召回后，是否比纯向量召回更容易组织宏观回答。
*   **自动化潜力**：哪些方向已经适合半自动聚类，哪些方向仍然需要人工主导。

### 5. 报告评估
对生成的设定报告重点看四点：

*   是否有清晰章节结构；
*   是否能回指到底层原文；
*   是否能标记冲突与不确定性；
*   是否真的帮助作者或设定党整理世界观。

## 九、 高危雷区与应对策略
1.  **“幽灵坐标”问题**
    *   **风险**：模型给出的定位信息不稳定，直接输出坐标容易幻觉。
    *   **应对**：模型输出 `start_anchor_quote` / `end_anchor_quote`，程序反查原文并给出定位置信度；无法稳定定位的记录不进入主索引层。
2.  **中心 Span 选错**
    *   **风险**：中心 Span 本身代表性不足，会把整簇带偏。
    *   **应对**：首版允许人工替换中心 Span，并记录替换历史；中心 Span 只是导航锚点，不作为唯一真相来源。
3.  **上下文扩展过度**
    *   **风险**：为回答宏观问题不断向外扩展原文，导致噪声淹没有效信息。
    *   **应对**：优先按章节边界、共享实体、主题相关性扩展，严格控制总 Token 预算。
4.  **单一路径召回失真**
    *   **风险**：只依赖图召回或只依赖向量召回，都会在宏观问题上留下明显盲区。
    *   **应对**：将图召回、向量召回、BM25 作为标准多路召回配置，并通过 Union + `Reranker` 统一收口。
5.  **聚合失真**
    *   **风险**：半自动聚类把不相干的 Span 拉进同一簇，导致图结构看起来漂亮但不可信。
    *   **应对**：聚类只在人工指定方向内进行；成员 Span 进入主簇前需经过规则过滤或人工确认。
6.  **精排失效**
    *   **风险**：候选池过大或精排模型不合适时，真正关键证据会被淹没，导致后续答案质量波动。
    *   **应对**：持续评估 `Reranker` 的 Top-K 命中率，并允许按问题类型调整候选池大小和精排阈值。
7.  **报告幻觉**
    *   **风险**：模型在生成长报告时，将若干合理猜测写成确定设定。
    *   **应对**：报告模板中显式区分“原文明确支持”“高概率推断”“待确认空白”三种结论等级。
8.  **工程复杂度失控**
    *   **风险**：过早引入图数据库、多 Agent、复杂聚类，导致系统难以维护。
    *   **应对**：坚持“先单书分析系统，后通用世界观平台”的演进顺序，每个阶段只引入一个新的复杂组件。
